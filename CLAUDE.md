# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

A portfolio-style AWS environment provisioned with **Terraform** and configured with **Ansible**. It models a small web stack: a VPC (public/private subnets, IGW, NAT), EC2 app instances in the public subnet, an S3 static website, and least-privilege security groups. There is no application code here — only IaC, configuration management, and CI/CD pipeline definitions.

## Repository Layout

- `terraform/bootstrap/` — One-time setup that creates the remote-state S3 bucket. Run before any environment. Has its own local state (`terraform.tfstate` is committed here).
- `terraform/modules/` — Reusable modules (`network`, `security_groups`, `ec2_app`, `s3_static_site`). Each wraps an official `terraform-aws-modules/*` module.
- `terraform/environments/{dev,prod}/` — Root modules; each is a **separate Terraform state** (S3 backend, key `environments/<env>/terraform.tfstate`). `main.tf` wires the modules together.
- `ansible/` — Roles (`docker_engine`, `webserver`) and play entrypoints. Inventory files are generated by Terraform, not hand-written.
- `.gitlab-ci.yml` + `.gitlab/ci/build_plan_provision_configure.yml` — The active CI/CD pipeline (parent + child).
- `azure-pipelines.yml` — Alternate Azure DevOps pipeline (currently `trigger: none`).
- `.github/workflows/terraform.yml` — GitHub Actions PR check; note its paths (`environments/<env>/ansible/...`) do **not** match the real layout and would fail as written.
- `Dockerfile` — Builds the `huddlesk/uv-ansible-terraform-alpine` CI runner image (uv + ansible + ansible-lint + Terraform 1.14.4).

## Common Commands

Terraform (run from inside an environment dir, e.g. `terraform/environments/dev`):

```shell
terraform init                                   # backend is configured in backend.tf
terraform plan  -out=tfplan                      # required vars: ami_id, unique_suffix, key_name
terraform apply tfplan                           # apply the exact reviewed plan
terraform fmt -recursive && terraform validate
terraform destroy
```

Required variables have **no defaults** and must be supplied (`-var=...`, a `terraform.tfvars`, or `TF_VAR_*` env vars): `ami_id`, `unique_suffix`, `key_name`. CI passes these as `TF_VAR_*`.

Bootstrap (run once, from `terraform/bootstrap/`):

```shell
terraform init && terraform apply               # creates the tf-state S3 bucket
```

Ansible (run from `ansible/`, after a Terraform apply has generated the inventory):

```shell
ansible-playbook -i inventory-dev.ini site.yml          # installs Docker (docker_engine role)
ansible-playbook -i inventory-dev.ini site-nginx.yml    # installs/configures Nginx (webserver role)
ansible-lint site.yml
```

## Architecture Notes / Gotchas

- **Inventory is generated, never edited.** `terraform/environments/<env>/main.tf` renders `ansible_inventory.tpl` via a `local_file` resource to `ansible/inventory-<env>.ini`. The generated host group is `[ec2_instances]` — all playbooks target that group. (`inventory-dev.ini` is gitignored/removed; do not rely on a checked-in copy. The README's references to `site-dev.yml` are stale — the real entrypoints are `site.yml` and `site-nginx.yml`.)
- **`ec2_app` is created with `count = 3`** in `environments/dev/main.tf`, so `ec2_public_ip` is a list and the inventory loops over all instances — despite docs describing "a single EC2 instance."
- **State locking uses the S3 native lockfile** (`use_lockfile = true` in `backend.tf`); the DynamoDB lock table is commented out in both `backend.tf` and `bootstrap/main.tf`.
- **Region is `us-east-1`** throughout (backend, CI `AWS_DEFAULT_REGION`). The dev `aws_region` var defaults to it but is not consumed by `main.tf` directly.
- **`ssh_cidr` defaults to `0.0.0.0/0`** — open SSH; tighten for any real deployment.
- Modules are thin wrappers around `terraform-aws-modules/{vpc,ec2-instance,s3-bucket}/aws`; check the module `main.tf` to see which upstream inputs are exposed.

## CI/CD Flow (GitLab — the active pipeline)

1. **`.pre` (generate-config):** computes the latest git tag, writes `generated-config.yml` pinning the runner image tag, and aborts if HEAD is already the latest tag (prevents redundant runs on `main`).
2. **trigger-child:** includes `.gitlab/ci/build_plan_provision_configure.yml` plus the generated config, passing the `environment` input (`dev`/`prod`).
3. Child stages: **build_runner** (build+push the Docker image, tags only) → **lint-and-plan** (`ansible-lint` + `terraform plan -out=tfplan`, artifacts the plan) → **provision** (manual; `terraform apply tfplan`) → **configure** (manual; pulls SSH key from GitLab Secure Files, runs `ansible-playbook`).

`provision` and `configure` are **manual** gates. The plan artifact from `lint-and-plan` is what gets applied, so re-running plan after review changes what deploys.

Runner image tag format: `MAJOR-UV-ANSIBLE-TERRAFORM` (e.g. `0-10.2-2.1-14.2`); built/pushed only on tagged commits.

## Conventions

- A new environment = a new `terraform/environments/<env>/` dir with its own `backend.tf` key, `variables.tf`, and `main.tf`. Add the env to the pipeline `options` lists in both GitLab spec files.
- Apply common tags via the `local.common_tags` map and pass `tags = local.common_tags` into every module call.
- Module changes propagate to all environments — `plan` both `dev` and `prod` after editing anything under `terraform/modules/`.
